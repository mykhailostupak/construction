# Construction Landing Page — Design Spec

**Date:** 2026-04-09
**Project:** `construction` (GitHub repo)
**Status:** Approved

---

## Overview

A static landing page for a small general contractor firm (residential & commercial). Primary purpose: display open job positions for hiring workers and subcontractors. Managed via Google Sheets, auto-synced via GitHub Actions, hosted free on GitHub Pages.

---

## Stack

| Layer | Technology | Notes |
|---|---|---|
| Frontend | HTML / CSS / JS | No framework, no build step |
| CMS | Google Sheets (public) | Readonly, no auth required |
| Hosting | GitHub Pages | Default URL → custom domain swappable via `CNAME` |
| Sync | GitHub Actions | Cron + `workflow_dispatch` |
| Translation | Gemini Flash API (free tier) | 1500 req/day, cached by content hash |
| Bot trigger | Telegram + Cloudflare Worker | Worker calls GH `workflow_dispatch` |

## Google Sheet Schema

One row per job. Columns in order:

| Column | Type | Description |
|---|---|---|
| `id` | string | Unique job identifier (e.g. `job-001`) |
| `active` | boolean | `TRUE` shows the listing, `FALSE` hides it |
| `title` | string | Job title in source language |
| `description` | string | Full job description in source language |
| `type` | enum | `permanent` / `monthly` / `daily` / `specific` |
| `rate` | number | Numeric rate value (e.g. `25`, `5500`, `180`) |
| `rate_type` | string | Display unit (e.g. `zł/h`, `zł/mo`, `zł/day`) |
| `location` | string | City or region (e.g. `Warsaw`) |
| `urgency` | string | Free text: `immediately`, `from now`, `from Apr 15`, `Apr 20–23` |
| `contact_option` | enum | `phone` / `email` / `phone+email` |

## Data Flow & Sync

### GitHub Action (`sync-jobs.yml`)

**Triggers:**
- Cron: `0 4,8,12,16 * * *` UTC (4 AM, 8 AM, 12 PM, 4 PM)
- `workflow_dispatch` (called by Cloudflare Worker on Telegram `/sync`)

**Steps:**
1. Fetch all rows from the public Google Sheet via Sheets API (HTTP GET, no auth)
2. Filter rows where `active = TRUE`
3. Load `data/translations.json` from repo (existing cache)
4. For each job: compute `hash(title + description)`
5. Collect all jobs where hash differs from cached value into a single batch
6. If batch is non-empty → **one Gemini Flash API call** with all new/changed jobs bundled in the prompt (e.g. JSON array of `{id, title, description}`) — response is a JSON array of translations, one entry per job. Parse and assign back by `id`. This avoids RPM limits from per-job calls.
7. Update `data/translations.json` with new translations and hashes
8. Build `data/jobs.json` with all fields + all language variants embedded
9. Commit both `data/` files to repo → GitHub Pages auto-deploys

### Secrets

**GitHub Actions repo secrets:**

| Secret | Used by |
|---|---|
| `GEMINI_API_KEY` | GH Action (translation step) |
| `GITHUB_TOKEN` | Auto-provided by Actions (repo write for committing `data/`) |

**Cloudflare Worker environment secrets** (set in Cloudflare dashboard):

| Secret | Purpose |
|---|---|
| `TELEGRAM_BOT_TOKEN` | Validate incoming Telegram webhook requests |
| `TELEGRAM_ALLOWED_USERS` | Comma-separated Telegram user IDs allowed to trigger sync |
| `GH_PAT` | Fine-grained PAT with Actions write scope → calls `workflow_dispatch` |

### Cloudflare Worker

~30 lines of JS. Free tier (100k req/day).

1. Telegram webhook POSTs to `https://your-worker.workers.dev/`
2. Worker validates `TELEGRAM_BOT_TOKEN` and user ID against `TELEGRAM_ALLOWED_USERS`
3. If `/sync` command → POST to GitHub API `workflow_dispatch` on `sync-jobs.yml`
4. Immediately replies to Telegram: "⏳ Sync started..."
5. GH Action runs; on completion it calls back via a second `workflow_dispatch` step that uses the Telegram Bot API to send the result message to the originating chat ID:
   - **Success:** "✅ Sync complete — {N} jobs updated, {M} translated."
   - **Error:** "❌ Sync failed — {error summary}. Check Actions log."

> **Implementation note:** The GH Action receives the Telegram `chat_id` as a `workflow_dispatch` input from the Worker. The final step of the Action sends the result message directly to Telegram via Bot API (`sendMessage`). This keeps the Worker stateless — no polling needed.

## jobs.json Structure

The GH Action combines the sheet's `rate` (number) and `rate_type` (string) columns into a single `rate` display string (e.g. `25` + `zł/h` → `"25 zł/h"`). Raw numeric values are not stored in `jobs.json`.

```json
[
  {
    "id": "job-001",
    "type": "permanent",
    "rate": "25 zł/h",
    "location": "Warsaw",
    "urgency": "from Apr 15",
    "contact_option": "phone",
    "title": {
      "en": "Framing Carpenter",
      "pl": "",
      "uk": "",
      "ru": ""
    },
    "description": {
      "en": "Experienced framer needed for residential projects. Must have own tools.",
      "pl": "",
      "uk": "",
      "ru": ""
    }
  }
]
```

## Page Structure

### Nav (sticky)
- Left: firm name placeholder (text, no logo)
- Right: language switcher — `EN` `PL` `UA` `RU` buttons
- Active language button highlighted in `--color-accent`
- Dark background with bottom accent border

### Hero
- Dark background (`--color-bg-hero`)
- Tagline: "We Build. We're Hiring."
- Subtext: "Residential & commercial projects."
- CTA button: "View Open Positions ↓" — smooth-scrolls to jobs section
- Bottom: 4px solid `--color-accent` separator line

### Jobs Section
- Background slightly different from hero (`--color-bg-jobs`) for visual separation
- Section label: "Open Positions" (uppercase, letter-spaced)
- **Grid breakpoints:**
  - `< 480px` → 1 column
  - `480px – 768px` → 2 columns
  - `> 768px` → 3–4 columns (`auto-fill`, `minmax(260px, 1fr)`)
- **Empty state:** dashed border card, icon, "No open positions right now" message

### Job Card
- Fixed `min-height` for grid uniformity
- Left border (4px) colored by job type (from `--color-type-*` vars)
- **Header row:** title (left, 18px bold, white) + rate (right, 17px, `--color-accent`)
- **Meta line:** `📍 location · type · urgency` (15px, muted color)
- **Description:** 15px, 1.6 line-height, muted color
- **Apply button:** centered, ~55% card width, `--color-accent` bg, black text, uppercase

### Apply Modal
- Full-screen overlay on Apply click
- Centered panel: job title, full description, contact block
- Contact block rendered by `contact_option`:
  - `phone` → "Call us: {phone}"
  - `email` → "Email us: {email}"
  - `phone+email` → both
- Contact values sourced from `config.js` (static, not in sheet)
- Close: X button or backdrop click
- Extensible: future additions (email form, WhatsApp) = new block inside modal

### Footer
- Firm name, phone, email, copyright year
- Dark background, minimal

PL/UK/RU files export same keys with empty string values — to be filled manually.

### Language switching

`i18n.js` holds `currentLang` state. On switch:
1. Update all `[data-i18n]` elements in the DOM
2. Re-render job cards with correct language variant from `jobs.json`
3. Persist selection to `localStorage`
4. Load saved preference on page init

## Code Quality Standards

- **HTML:** semantic elements (`<nav>`, `<main>`, `<section>`, `<article>`, `<footer>`), `data-i18n` attributes for translatable text, `data-job-id` on cards
- **CSS:** CSS custom properties for all design tokens, BEM-inspired class names (`.job-card`, `.job-card__title`, `.job-card__meta`), mobile-first media queries, no inline styles
- **JS:** `const`/`let` only, descriptive function names (`renderJobCards`, `openApplyModal`, `switchLanguage`, `fetchJobs`), error handling on all fetch calls with user-facing fallback message, no global state pollution
